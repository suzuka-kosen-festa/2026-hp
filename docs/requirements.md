### 3.3 booth 詳細仕様

**構造**
- 個別詳細ページなし。カード一覧のみで完結
- 第1層＝タブ（カテゴリ切り替え）: 出店 / 学科展示 / イベント / ライブ
- 第2層＝タグ絞り込み（複数選択・OR条件）。タブごとに意味の異なるタグセットを表示。**絞り込みバーはスクロール時も画面上部にsticky固定**
  - 出店タブ: 飲食(フード/スイーツ) / レク / 物販 / 展示
  - 学科展示タブ: M科 / E科 / I科 / C科 / S科（1学科あたり約6件、計30件規模）
  - イベントタブ / ライブタブ: `timetable.json` を再利用してカード化（timetableページとデータソース共通）

**データ（`booths.json`）**
- 「出店」「学科展示」は1本の`booths.json`に統合（`category`フィールドで区別）。「イベント」「ライブ」は`timetable.json`をそのまま参照するため`booths.json`には含めない
- フィールド: `id` / `category`（出店 or 学科展示） / `name` / `group`（団体名・学科名） / `description`（文字数上限あり、超過分はCSS `line-clamp`で省略。アコーディオン展開はなし） / `tags`（タブごとの第2層タグ） / `location`（テキスト。基本は3.4の7エリア表記、出店のうちエリア③④は詳細位置を別途持つ） / `image`（任意。nullの場合はUIで「NO IMAGE」プレースホルダーを表示しカード高さを揃える）
- 並び順: ソートロジックなし。JSON配列の記載順をそのまま表示順とする（委員会側が意図的な掲載順を制御できるようにするため）
- 両日(10/31・11/1)出店が前提。片日のみ出店するブースは無いため`day`系フィールドは持たない
- フリーテキスト検索: なし（タブ×タグの絞り込みのみ）
- 表示件数: ページネーションなし。全件1ページ表示 + 画像は`loading="lazy"`
- 更新運用: 非エンジニアの直接編集ではなく、**エンジニアがJSONを編集してpush**する運用（§6参照）

**常設企画「コラージュカメラ(仮)」の特別扱い**
- 概要: お題で指定された物・場所の写真を複数枚撮影し、1枚のフォトコラージュを作るWebアプリ企画。会場内どこでも各自のスマホからアクセス可能（物理スポットに紐づかない）
- テーマ「collage」を体現する目玉企画のため、他の常設企画と同列ではなく**特別扱い**とする:
  - `/booth` 常設セクションの最上部に固定表示。紹介用サンプル画像＋概要文＋Webアプリへのリンクボタンのみ（お題の内容はサイト側に出さず、アプリ内で見せる）
  - home にも「contentsカード4枚」直後に**横長プロモーションバナー**を追加設置（3.1参照）。サンプル画像＋短い紹介文＋リンクボタンの横長版
- `booths.json`/`timetable.json`のいずれのスキーマにも無理に当てはめず、単発の特別枠として扱う（1件のみで拡張性を要しないため、非エンジニア運用を前提としたデータファイル化はせず、コンポーネント内で直接管理してよい）

### 3.4 map 詳細仕様（booth連携含む）

会場マップは画像+凡例の静止表示ではなく、**エリア単位でタップ可能なインタラクティブマップ**とする（個別ブースへのピン配置はしない＝運用コスト過大のため）。

7エリア構成とタップ時の遷移先:

| No | エリア名 | 場所 | タップ時の遷移先 |
|---|---|---|---|
| ① | LiveStage | 第一体育館 | `/timetable`（該当ステージへディープリンク） |
| ② | MainStage | 屋外 | `/timetable`（該当ステージへディープリンク） |
| ③ | 飲食バザー | 屋外 | エリア詳細マップ（具体的な店舗配置を表示） |
| ④ | 展示・レク | 教室棟 | エリア詳細マップ（具体的な店舗配置を表示） |
| ⑤ | ゲームイベント | 教室棟 | `/booth`（出店タブへディープリンク） |
| ⑥ | SubStage | 屋外 | `/timetable`（該当ステージへディープリンク） |
| ⑦ | 学科展示 | 第二体育館 | `/booth`（学科展示タブへディープリンク） |

- 全ページ共通で`location`はこの7エリア表記を基本とする。ただし③④の出店は「エリア＋具体的な位置」を追加で持つ
- ③④のエリア詳細マップに使う座標データ（区画番号・配置座標）は**未入手（実行委員会の配置班からの提供待ち）**。入手までは位置をテキスト表示（例:「Aブロック3番」）に留め、ピン表示は保留（§8参照）
- 遷移はページ遷移のみでなく、**タブ選択・タグ絞り込み・スクロール位置をURLパラメータ等で引き継ぐディープリンク**として実装する（例: マップ⑤タップ → `/booth`に遷移し「出店」タブが選択された状態で該当セクションまでスクロール）

**ベース画像・レイアウト**
- ベース画像はFigmaでベクター作成中（世界観との整合は別途デザイン検討課題）
- ビジュアルなインタラクティブマップに加えて、**7エリアのテキストリスト（凡例）を併設**し、どちらからでも同じ遷移ができるようにする（アクセシビリティ・視認性の補完）
- マップの向き（横長維持 or 縦長化）とテキストリストの配置は、レイアウトそのものなのでFigma側で決定する（コードファーストで先に決めない）
- SPはピンチズーム/パン非対応。固定表示（画面幅にフィット）で十分とする

**インタラクション**
- エリアタップ時は一瞬ハイライトする視覚フィードバックを入れてから遷移する
- エリア③④（飲食バザー・展示・レク）は、座標データが揃うまでタップ時に「準備中」表示のみとする（データ到着後にビジュアル詳細マップへ差し替える開発タスクが発生する。§8参照）
- ①LiveStage・②MainStage・⑥SubStageの3エリアには、現在進行中の企画があればタイトルを表示（長い場合は省略）。判定ロジックは3.5のtimetable「NOW」バッジと共通化。⑤ゲームイベント・⑦学科展示は常時開催のため対象外

### 3.5 timetable 詳細仕様

**構造**
- グリッド表ではなく、**ステージ別にグルーピングした縦の時系列リスト**
- 第1層＝日タブ（Day1 / Day2）、第2層＝ステージ/カテゴリのタグ絞り込み（LiveStage / MainStage / SubStage / ゲームイベント 等、複数選択・OR、boothと共通のUIコンポーネントを流用）
- 「常設」企画は時系列リストに混ぜず、日タブ上部に**別枠の常設セクション**として両日共通表示
- 現在時刻に対応する企画には**「NOW」バッジ**を表示（自動スクロールはなし）

**データ（`timetable.json`）**
```json
{
  "id": "一意のスラッグ",
  "day": "day1 | day2",
  "location": "LiveStage | MainStage | SubStage | ゲームイベント | ...",
  "start_time": "10:00",
  "end_time": "10:30",
  "title": "企画名・バンド名",
  "group": "団体名・出演者名（任意）",
  "special_tag": "中夜祭 | 決勝バンド | null"
}
```
- `location`は`booths.json`と用語を統一（`stage`ではなく`location`）
- 決勝バンドは`location: MainStage` / `day: day2` / `special_tag: 決勝バンド`（MainStageのDay2セクションに掲載）
- 中夜祭は`location: LiveStage` / `day: day1` / `special_tag: 中夜祭`（LiveStageセクション内、時刻順で夜に自然配置）
- `/booth`のイベント/ライブタブは、この`timetable.json`の`day`（イベント用）／`special_tag`+`day`（ライブ用）をタグとして絞り込みに使う

### 3.6 access 詳細仕様

- アクセス手段（電車・バス・徒歩）は§1の情報のまま。埋め込み地図は使わず、外部リンク（Googleマップで開く）ボタンのみ設置
- 駐車場は**主従関係**: メイン＝鈴鹿医療科学大学白子キャンパス、予備＝MEGAドン・キホーテUNY鈴鹿店（立体駐車場3F〜7F、利用時は上の階から順に駐車）。シャトルバスなし
- ドン・キホーテ駐車場への経路案内は、既存の案内資料（暫定素材。後日世界観に合わせて差し替えの可能性あり）を活用する:
  - 経路図（矢印・道路のグラフィック部分）は画像のまま流用し、「貼り付けた紙片」演出（回転+ハードシャドウ、design-system.md参照）で見せる
  - 手順テキスト（①②③④、駐車順序の注意書き等）は画像に埋め込まず、サイト自前のタイポグラフィで打ち直す
- ページに**「最終更新日」を表示**し情報の鮮度を示す（`access.json`に`updated_at`等を保持）

### 3.7 news 詳細仕様

- 基本は「リンク集（ポータル）」。一覧は**タイトルのみ**（日付＋NEWバッジ）で、画像表示は今回見送り（将来必要になったら追加検討）
- リンク先は柔軟に持たせる: 既存ページへのディープリンク（例: 出店一覧公開のお知らせ→`/booth`）が基本。単体で読ませる内容がある場合のみ`/news/[id]`記事ページを生成する
- データは`news.json`1本で管理。個別記事用のMarkdown Content Collectionsは今回作らない（YAGNI。本文が必要な記事が増えたら再検討）
- NEWバッジは**日付からの自動判定**（例: 公開後7日以内は自動表示。閾値は実装時の定数として調整可能）。手動フラグは廃止

**データ（`news.json`）**
```json
{
  "id": "一意のスラッグ",
  "date": "2026-09-15",
  "title": "お知らせタイトル",
  "href": "リンク先（内部ディープリンク or /news/[id]）",
  "body": "Markdown本文（任意。ある場合のみ/news/[id]ページを自動生成）"
}
```

### 3.8 sponsors 詳細仕様

- 法人協賛・個人協賛を**2つのリストに分けて**それぞれ**50音順・名前のみ**で表示（ロゴ・ランク表示なし）
- `featured: true`の高額協賛企業のみ、**フッター（全ページ共通）でロゴを表示**する（§3.1の「主要協賛企業」に対応）

**データ（`sponsors.json`）**
```json
{
  "name": "企業名・個人名",
  "type": "corporate | individual",
  "featured": false,
  "logo": "画像パス or null（featured: trueの場合のみ使用）"
}
```

## 4. 共通要素

- **背景**: コラージュ（テープや紙が切り貼りされた質感）。実装は design-system.md 参照
- **ヘッダー**:
  - PC（約900px以上）: 左にロゴテキスト、右に7項目横並びnav。hover/現在ページに赤テープ下線（Figmaデザイン準拠）
  - SP: 細い固定バー（左: 小ロゴ、右: ハンバーガー=テープ3枚）→ タップで全画面コラージュメニュー
- **フッター**: home 最下部と同一内容を全ページ共通化
- （検討中）SP画面下固定のクイックバー（map / timetable / booth）— 当日利用のショートカット

## 5. OPムービー（スプラッシュ型）

- 初回訪問時のみ全画面再生。`sessionStorage` で2回目以降スキップ
- `autoplay muted playsinline`（スマホ自動再生の必須条件）
- **スキップボタンを常時表示**（再生開始直後から）
- 再生失敗・reduced-motion・低速回線時はロゴ静止画ヒーローにフォールバック
- 素材: OP.mp4（9.8秒 / 1900×906 / 8.5MB / H.264+AAC）
  - 再エンコード目標: H.264 2〜3MB + 可能なら WebM(VP9/AV1) 併用、SP向け720p版
  - 音声トラックは削除（muted必須のため不要）

## 6. コンテンツ管理（データ駆動）

コンテンツはコードから分離しデータファイル（JSON）で管理する。**更新作業は非エンジニアの実行委員が直接行うのではなく、エンジニアがJSONを編集してpushする運用**とする（当初「非エンジニアでも更新できる形」を想定していたが、実行委員のGit操作は現実的でないため方針転換）。

| データ | 形式 | 更新頻度 |
|---|---|---|
| news | JSON（リンク集ベース。`body`があれば記事ページ自動生成。詳細は3.7） | 開催前〜当日 高 |
| booths.json | JSON（出店・学科展示。詳細は3.3） | 確定後ほぼ固定 |
| timetable.json | JSON（日・時刻・location・企画名・special_tag。`/booth`のイベント/ライブタブと共用。詳細は3.5） | 確定後ほぼ固定 |
| sponsors.json | JSON（名称・type・featured・logo。詳細は3.8） | 随時追加 |
| access.json | JSON（駐車場情報・`updated_at`。詳細は3.6） | 変更時のみ（高頻度ではないが重要） |

- 更新フロー: GitHub上でエンジニアが編集 → push → Cloudflare Pages 自動ビルド・デプロイ
- スキーマは実データのサンプルをもらってから確定する

## 7. 技術・非機能要件

- **Astro**（静的生成）+ 素のCSS を土台に、インタラクション/アニメーションが必要な箇所のみ React + Framer Motion（`motion`）を「アイランド」として追加（例: ハンバーガーメニュー開閉、セクションのスクロール登場演出）。それ以外のJSは booth 絞り込み・OP制御のみ
- **Cloudflare Pages** にデプロイ（GitHub連携の自動ビルド）
- パフォーマンス: 4G回線で LCP 2.5秒以内を目安。画像は圧縮 + `loading="lazy"`、OPムービーはスプラッシュ以外でプリロードしない
- OGP画像・favicon（ロゴ流用）・タイトル/description 整備
- アクセシビリティ: 文字コントラスト確保、画像alt、reduced-motion 対応（マーキー・OP停止）
